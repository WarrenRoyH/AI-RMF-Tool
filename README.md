# AI-RMF Lifecycle Tools

![CI Status](https://github.com/WarrenRoyH/AI-RMF-Tool/actions/workflows/ci.yml/badge.svg)
![Coverage](https://codecov.io/gh/WarrenRoyH/AI-RMF-Tool/branch/main/graph/badge.svg)

An automated, persona-driven toolkit designed to operationalize the **NIST AI Risk Management Framework (AI RMF) 1.0**. 

This project provides a modular suite of open-source agents that handle the **Govern, Map, Measure, and Manage** functions of the NIST framework, ensuring your AI applications are secure, transparent, and compliant.

---

## Core Design Philosophy: Zero-Friction UX
This toolkit is built on the principle that **compliance should not require a PhD in AI or Cybersecurity**. The guiding directive for this application is extreme user-friendliness. Anyone, technical or not, should be able to deploy, configure, and understand the output of these tools. 

All future development, including autonomous optimization by agents, must prioritize:
1. **Plain Language:** Avoiding overly dense technical jargon in logs, terminal outputs, and reports.
2. **One-Click Deployments:** Keeping the setup process as close to "Zero-Install" as possible.
3. **Intuitive Personas:** Ensuring the agents (Librarian, Sentry, Auditor) communicate and act in ways that are easily understood by non-technical stakeholders.
4. **Hybrid Interface Architecture:** Both a robust Command Line Interface (CLI) and a local-first Graphical User Interface (GUI, e.g., via a local `index.html`) must be available. Users should never be forced to choose one over the other; all core functionality must be accessible through both paradigms to ensure maximum accessibility for all skill levels.

## 1. What It Does
The AI-RMF Lifecycle Tools transform high-level NIST governance principles into technical reality. The toolkit provides:
*   **Interactive Governance:** A "Librarian" persona that interviews stakeholders to generate an AI-BOM and risk manifest.
*   **Vulnerability Scanning:** An "Adversary" persona that probes models for prompt injection and jailbreaks.
*   **Real-time Protection:** A "Sentry" persona that acts as a firewall for LLM inputs and outputs.
*   **Continuous Observability:** An "Auditor" persona that tracks model drift and policy violations in production.
*   **Automated Red-Teaming:** A "Red Teamer" persona for CI/CD stress-testing of bias and accuracy.

---

## 2. How It Works: The Persona Lifecycle
The toolkit follows a **"Librarian-First"** workflow. Human-centric governance (Govern) creates a machine-readable manifest that automatically configures the technical agents (Map, Manage, Measure).

1.  **Step 0: The Architect (Deployment)** – An interactive questionnaire determines your environment (Cloud, Hybrid, or On-Prem) and configures the optimal security sandbox.
2.  **Step 1: The Librarian (Govern)** – Conducts a NIST-aligned interview to define the project’s **AI-BOM**, safety policies, and risk thresholds.
3.  **Step 2: The Adversary (Map)** – Automatically scans your model using **Garak**, targeting the specific risks identified by the Librarian.
4.  **Step 3: The Sentry (Manage)** – Deploys **LLM Guard** middleware to enforce your "No-Go" policies (e.g., blocking PII or medical advice) in real-time.
5.  **Step 4: The Auditor & Red Teamer (Measure)** – Uses **Arize Phoenix** for local log analysis and **Promptfoo** for automated regression testing against your benchmarks.

## 3. Application Architecture

### Universal Portability with `uv`
The toolkit uses the **`uv`** Python engine to ensure a "Zero-Install" experience.
*   **No Python Setup:** `uv` automatically manages the correct Python version and isolated virtual environment (`.venv`) without touching your system Python.
*   **Fast & Light:** 10x faster than traditional tools, with a single static binary.

### Security Isolation (The Sandbox)
To ensure agents cannot access sensitive host data, the toolkit utilizes OS-native sandboxing:
*   **Linux:** Uses `bubblewrap (bwrap)` to jail processes to the `workspace/` folder.
*   **macOS:** Uses `sandbox-exec` with custom security profiles.
*   **Cloud/On-Prem:** Fully compatible with unprivileged environments (no `sudo` required).

### High-Level Blueprint
```text
ai-rmf-tools/
├── ai-rmf (CLI Launcher)     # The entry point for all personas
├── .env.example             # Template for LLM API keys
├── workspace/               # MOUNTED VOLUME (The only folder agents can touch)
│   ├── project-manifest.json # Generated by Librarian (The Source of Truth)
│   ├── logs/                # Production traces for the Auditor
│   └── reports/             # Security scan results
├── core/                    # Unified LLM Provider & Sandbox logic
└── sandbox/                 # Security profiles (bwrap/macos)
```

## 4. How to Use It

### Prerequisites
*   A terminal (bash, zsh, or sh).
*   No Docker or Python installation is required (handled automatically by `uv`).

### Step 0: Initialize & Deploy
Run the Architect wizard (via bootstrap.sh) or check your environment:
```bash
./ai-rmf health
```
*The health check verifies workspace structure, library dependencies, and API connectivity for both primary and test model pools.*

### Step 0: Initial Setup (The Wizard)
New to AI-RMF? Use the intent-based setup wizard to scaffold your project manifest in seconds.
```bash
./ai-rmf setup
```
*The Librarian will ask for a brief description of your project and generate a NIST-aligned manifest draft.*

### Step 1: Start Governance (The Librarian)
Generate your NIST AI-RMF manifest:
```bash
./ai-rmf govern
```

### Step 2: Scan for Vulnerabilities (The Adversary)
Run a security probe against your target model:
```bash
./ai-rmf map --target "http://localhost:11434/v1"
```

### Step 3: Monitor Production (The Auditor)
Launch the local observability dashboard:
```bash
./ai-rmf measure --mode audit
```

### B. Technical Architecture Patterns
These patterns define the **Sentry's** injection points and the **Adversary's** probe surface.

*   **Pattern A: The RAG (Retrieval-Augmented Generation) Pipeline:** Web UI → API Gateway → Vector DB → LLM. *Key Risk: Indirect Prompt Injection from poisoned data sources.*
*   **Pattern B: The Agentic Orchestrator (Tool-Calling):** User → Agent (LangChain) → External Tools (Python/SQL). *Key Risk: Remote Code Execution (RCE) via malicious tool arguments.*
*   **Pattern C: The High-Throughput Batch Processor:** S3/Kafka → LLM Worker Cluster → Database. *Key Risk: Large-scale data poisoning and latency-based Denial of Service.*
*   **Pattern D: The Federated Edge Deployment:** Mobile App (Local SLM) → Aggregator → Central Cloud. *Key Risk: Model inversion and local PII exposure in edge logs.*
